How should we structure project management in Obsidian?

Three approaches with different trade-offs on simplicity, power, and maintainability

A

Bases-First (Recommended)

Each issue is a note. Bases provide the views.

Issues are plain Markdown files with frontmatter properties (status, priority, project, due date, parent). Obsidian Bases (`.base` files) query these notes and display them as tables, grouped by status or project. No community plugins needed.

Pros

  • Built entirely on core Obsidian features (Bases, properties, wikilinks)
  • Each issue is a full note — rich descriptions, links to docs vaults, sub-tasks via wikilinks
  • Bases give you table/list/card views with filters, grouping, and formulas out of the box
  • Easy to extend — add a new property and it just works

Cons

  • No true kanban board view (Bases doesn't have a board/column layout)
  • Many files over time — need good folder structure and archive strategy
B

Kanban Plugin + Properties

Issues are notes. A Kanban plugin provides the board view.

Same note-per-issue structure, but add the community Kanban plugin for drag-and-drop board views. Properties still drive metadata. You get visual boards AND Bases table views.

Pros

  • True kanban board with drag-and-drop between columns
  • Visual, tactile workflow that feels closest to Linear
  • Can combine with Bases for table views too

Cons

  • Depends on community plugin (maintenance risk)
  • Kanban plugin stores board state in its own format — can drift from properties
  • Two sources of truth for status (board position vs. frontmatter)
C

Flat Task Lists in Project Notes

Each project is one note. Tasks are checkbox items inside it.

Instead of one-note-per-issue, keep tasks as checkbox list items within a project note. Use the Tasks plugin or Bases task queries to aggregate across projects.

Pros

  • Simplest to set up — fewer files, less structure
  • Everything about a project in one place

Cons

  • Tasks can't have rich descriptions, sub-tasks, or their own metadata easily
  • Doesn't scale — project notes become unwieldy with 50+ tasks
  • Loses the "issue as a document" model that makes Linear powerful
  • Hard to link from other vaults to specific tasks
